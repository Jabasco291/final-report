\section{Research design}\label{sec:research-design}
% Research: overall approach, methods (e.g.\@ quantitative, qualitative, design science), rationale. Omit for pure system-design projects.

This project presents a comparative evaluation across three state-of-the-art candidate methods: S-Graphs+\cite{s_graphs_plus}, Hydra\cite{hydra} and Clio\cite{clio}. 

The methods were evaluated over five executions on a controlled and consistent simulation environment. This highest-performing method was determined through evaluation of three main categories: Metric accuracy, Semantic accuracy and Practical feasibility. This chosen system was then deployed on real-world sensor data of the \gls{hccr} laboratory.

Each method was deployed on the same simulated environment. The sensor data was recorded in a single rosbag using a \acrshort{rgbd} camera (Hydra and Clio) and 3D \acrshort{lidar} sensor (S-Graphs+) which were attached to a differential drive robot and teleoperated around the scene. The methods resulted in a 3D scene graph, which was then saved in JSON format for further statistical evaluation. 
Simulation results were evaluated quantitatively to determine the highest-performing statistically, then the real-world results were evaluated qualitatively. This is because ground-truth is accessible for simulation environments like Gazebo, however generating ground-truth can be complicated for real-world data.

\section{Specification}\label{sec:specification}
% what should they need to do (what requirements assessing)

To evaluate the suitability of the candidate methods for architectural tasks, requirements are categorised into Mandatory and Desirable requirements. These define the fundamental capabilities for \gls{slam} operations and the ideal capabilities for higher-level architectural tasks such as inventorying.

\subsection{Mandatory Requirements}\label{sec:spec:mandatory}

\begin{itemize}
    \item \textbf{Hierarchical Scene Representation:} The method must generate a 3D scene graph with multiple abstraction layers to provide architectural context.
    \item \textbf{Spatial Segmentation:} The method must be able to segment environments into distinct room nodes.
    \item \textbf{ROS Compatibility:} The method must be compatible with \gls{ros} input data for processing simulation data.
    \item \textbf{Operational Stability:} The method must be able to process a five-minute office scene without fatal errors.
    \item \textbf{Quantitative Evaluation:} The evaluation framework must calculate Recall, Precision, and F1-score for room nodes and object nodes where applicable.
\end{itemize}

\subsection{Desirable Requirements}\label{sec:spec:desirable}
\begin{itemize}
    \item \textbf{Semantic Object Layer:} The system should provide a semantic object layer to support automated inventorying or interactive tasks.
\end{itemize}

\section{Candidate Methods}\label{sec:methods}
% how each works and differ

The three candidate methods are S-Graphs+\cite{s_graphs_plus}, Hydra\cite{hydra} and Clio\cite{clio}. Below is a brief explanation of their main characteristics and why they were chosen for the evaluation.

S-Graphs+ takes 3D \acrshort{lidar} point cloud as input. It then analyses the point cloud to identify geometric planes, which are then used to detect rooms. The hierarchical structure consists of Keyframes, Walls, Rooms, Floors. There is no object segmentation or semantic labelling, its output mostly contains navigation and structural information. This method was selected to observe how its purely geometric structure would compare to methods which also incorporate semantics.

Hydra adopts a spatial approach using \acrshort{rgbd} data to reconstruct a 3D metric-semantic mesh of the scene. It uses a \gls{gvd} to represent a free-space graph to represent navigable spaces. This graph is then dilated to expose connectors such as doorways which allows clusters of nodes to be segmented into rooms. Hydra differs from S-Graphs+ by including an object layer, as well as using a neural network to assign semantic labels throughout the graph. This method was included to evaluate the spatial approach compared to geometric and task-driven approaches.

Clio takes a different approach to the other candidate methods. Instead of being task-agnostic, meaning it captures everything about the scene, Clio adopts a task-driven methodology. It takes a set of tasks and regions in natural language as input, then segments the scene into task-relevant objects. Clio’s semantics are open-set compared to Hydra’s closed-set semantics, meaning it doesn’t have a fixed set of possible semantic labels to choose from, instead they are decided upon depending on the tasks required. To identify regions (rooms), Clio groups semantic objects into regions rather than using geometric boundaries like the other 2 candidates. For example, a kettle and cooker are grouped into a kitchen. This method was chosen to evaluate how task-driven approaches compare to task-agnostic methods.

\section{Experimental Setup}\label{sec:design}

\subsection{Simulation}\label{sec:sim-setup}

The custom dataset used for the evaluation was the office scene shown in Figure \ref{fig:office}, recorded in Gazebo using a differential drive robot shown in Figure \ref{fig:robot}. The robot was configured to mimic the sensors on a Reachy2 \cite{Reachy2} robot, which represents a realistic candidate for indoor tasks due to an omnidirectional base and gripper hands. It had a velodyne 3D \acrshort{lidar} sensor (for S-Graphs+) attached to the base with a \acrshort{rgbd} camera (Hydra and Clio) attached to the body halfway up. The camera was a simulated camera setup to emulate the Intel Realsense D435 camera due to it being popular in robotics research, recording 1280x720 image at 30Hz, with 1.5708 Field of View. The office scene was sourced from \cite{gazebo-worlds-dataset}, which was segmented to include only a portion of the environment shown in Figure \ref{fig:office}. The decision to use a portioned section of the scene was made to ensure the recording was a feasible length whilst including multiple room types and objects.

Each method was deployed on the same rosbag recording, this made the evaluation fair, as it ensures they all process the same data, which allows for a direct comparsion.

Before testing commenced on the custom dataset, the methods were verified against example/known data to ensure they had correct behaviour. Hydra and Clio were tested on their respective example datasets. However, there was no provided example data for S-Graphs+, but it was deployed on a small recording of an indoor house from a prior Introduction to Computational Robotics module, which provided a simple environment to assess expected behaviour. Once expected behaviour was confirmed, the evaluation stage could then begin.

A fundamental part for deploying Clio requires a task/region list, which represents the tasks and regions it should be searching for. It was fed with Tasks: --- and Regions: ---. These were chosen to cater for all room types, as well as objects which are distinct and regular throughout the scene which would allow for detection analysis.

A key deployment limitation was observed with Clio. The rosbag image data had to be downsampled and played at maximum 0.25x speed in order for frames to be processed correctly and the system not to crash. Even with this input, GPU usage was observed to spike to 98\% occasionally, which suggests the bottleneck might have been to do with the hardware gap (SPECS) to (SPECS). Notably, Clio ran without issue on the example dataset at real-time playback, suggesting the limitation is a specific response to the custom dataset. This represents a real-world capability limitation, as Hydra was able to run in real-time playback with the original image size without issue.

\subsection{Real-World}\label{sec:real-setup}

The simulation phase concluded that Hydra was the best-performing method as an all round system (discussed in Section \ref{sec:res:best-method}). Therefore, this meant 3D \acrshort{lidar} data wasn’t required for the real-world deployment due to S-Graphs+ being excluded from this stage.

The ZED camera was the optimal choice for recording the scene due to its simple setup and \acrshort{rgbd} capabilities alongside pose estimates, it was set to record 720p quality at 30Hz, which enabled sufficient quality and coverage. To mimic the perspective of a traversal robot like Reachy2, The camera was handheld at torso height (approx. 1.2m) whilst being walked slowly around the HCCR lab.

As the ZED camera naturally lacks ROS message outputs, the svo2 formatted video was extracted into \acrshort{rgb} and Depth frames along with pose history of the robot. This data was then encoded into a \acrshort{ros}2 bag recording which can be processed by Hydra.

\section{Implementation}\label{sec:implementation}
% how deploy in real-world, what tech

\subsection{Simulation Environment Setup}\label{sec:sim-env-setup}

Each method requires different \acrshort{ros} environments. There was no \acrshort{ros} version which was supported by all three methods simultaneously. To ensure a smooth deployment of all three methods, Docker was used to help maintain three separate environments.

S-Graphs+ was installed natively with \acrshort{ros}2 Humble as it was the most lightweight of the candidate methods and didn’t require any major dependencies like CUDA. To ensure a smooth installation and integration to a custom dataset, the instructions from \cite{sgraphs-install} were followed.

Hydra was run through a Docker container with \acrshort{ros}2 Jazzy sourced from the official Hydra-ROS github \cite{hydra-ros-docker}. This setup allowed smooth running in parallel with S-Graphs+ as it removes \acrshort{ros} version conflicts. The Docker also allowed smooth operation due to the pre-installed dependencies for things such as CUDA and TensorRT, avoiding manual dependency configuration.

The original deployment system was a personal laptop with AMD Ryzen 5000 Series with AMD Radeon Graphics, a problem arose when deploying Hydra where only coordinate frames were updating, with no graph being built. This was identified for a lack of NVIDIA graphics to run TensorRT and CUDA. Once identified, deployment was moved to a \gls{hccr} lab computer which had more computing power (see Table \ref{tab:hardware_specs} for full details).

Clio was deployed using a \acrshort{ros} Noetic Docker sourced from \cite{cliodocker}. The decision to use a Docker was more important for this method due to the fact it needs \acrshort{ros}1 instead of \acrshort{ros}2. Whilst trying to build the workspace, an issue was identified which caused build failures. It was caused due to Clio being a \acrshort{ros}1 system, when some core dependencies had switched their default branches to \acrshort{ros}2, causing incompatibility failures. This was a known problem discussed in a community forum, which led to the discovery of a community-built container which explicitly uses the old \acrshort{ros}1 version of dependencies.

The setup of one native and two Dockerised methods enabled smooth transition between running each method,. By only changing the active environment rather than reinstalling \acrshort{ros} versions, this allowed each method to run under intended conditions. This made the deployment process more practical given the time constraints of the project.

\subsection{Robot Configuration and Data Recording}\label{sec:sim-robot-config}

The robot configuration used to record the custom dataset was sourced from \cite{mobile-3d-lidar-sim}, which contains a differential drive robot with a Velodyne 3D LiDAR sensor, which was the baseline platform for data collection. 

As only S-Graphs+ uses LiDAR data and the other methods use RGB-D, the URDF configuration was modified to add a camera on a pole. The camera specification was set to be similar to a Realsense D435 which is widely used in robotics research, it was attached around 1.2m up the pole to simulate torso height perspective. The robot was setup to emulate the Reachy2 robot \cite{Reachy2} style which is a realistic system that could be used for architectural tasks like building visualisation.

The robot was then teleoperated around the office scene to capture the rosbag data. This allowed control over the scene coverage and trajectory as well as simulating a realistic scenario for building visualistion.

The resultant custom office rosbag was a 317s recording, which was long enough to explore multiple rooms of different sizes (to assess room segmentation). The topics recorded in the rosbag are shown in Table \ref{tab:rosbag-topics} below.

\subsection{Method parameters}\label{sec:sim-method-params}

\noindent \textbf{Hydra}

The original Hydra parameters were not setup for a scene with such varying room-types like  the custom office scene, where small offices or bathrooms neighbour a large office space. This led to a massive under-segmentation of the scene due to dilation distance being almost the same size as these smaller rooms. For example, the uhumans2 parameters include a 0.5m and 1.2m min and max dilation respectively, along with a 10 min room size.

A variety of parameters were tested to try and fit the behaviour to the custom office scene structure. After testing six sets of parameters, it was decided to stick with the parameters found in reachy\_house.yaml. The most notable are shown in Figure \ref{fig:hydra-params}. 

\begin{figure}[H]
    \centering
    \includegraphics[width=0.5\textwidth]{figures/hydra_params.png}
    \caption{Snapshot of main Hydra parameters which were tuned to better suit the custom office scene.}
    \label{fig:hydra-params}
\end{figure}

A min and max dilation of 0.15 and 0.3 respectively was chosen to ensure the smaller rooms do not get swallowed when the graph of places is dilated as discussed in \ref{subsec:hydra}. 

Min room size was set to 5, which lowers from larger scenes like uhumans2, this was chosen to give the smaller rooms the best chance to be instantiated rather than being swallowed by the large open office. 

The plateau ratio defines how strict the system is about committing to segmentation earlier. This was lowered in order to encourage creating rooms before they get merged into the large space.

\noindent \textbf{Clio}

At the beginning, Clio was struggling to segment the different rooms and had a dictator region behaviour (where every region would be merged into a large node). This was likely due to a mismatch in between the scenes Clio was originally tuned for compared to the custom office scene.

Iterative testing helped to tune the Clio parameters to better suit the custom office scene. The final values were chosen based on observed improvements to segmentation and detection behaviour.

Voxel size was increased from 0.10m to 0.15m. This traded reconstruction quality to reduce the likelihood of objects being represented as fragmented instances. 

The FastSAM confidence threshold was lowered to allow more candidate masks to be accepted per frame. In a dataset with varying coverage of different regions due to size, like a smaller bathroom, this helps to give a greater chance to produce detection that reach the \acrshort{clip} embedding stage.

The score threshold for region detection was reduced in a similar way. This allowed regions that are less-visited or covered resulting in smaller embedding similarity scores to be recognised.

Object cluster size was adjusted to filter out small masks, which are likely noise and irrelevant views, and large masks, which occur when background regions merge into a single cluster.

These Clio parameters ended up prioritising recall over precision, this is useful in architectural tasks as its better for a region to be over-detected than missed.


\subsection{Real-World Data Pipeline}\label{sec:real-data-pipeline}

To record the HCCR lab for real-world data deployment, a handheld ZED camera powered by a NVIDIA Jetson AGX Orin was used. This recording method was chosen due to its portability and ease of setup. The recording was then parsed into RGB and Depth frames with pose estimates (using code provided by PhD Student Emin Safa Tok). The frames were then converted into a ros2 bag using a conversion script. Generative AI Claude (Anthropic) \cite{Claude2024} was used to help develop this script, justification is outlined in Section \pageref{ai-use}. The script takes the data and sequentially writes them as rosbag topics . This rosbag could then be processed by the best-performing method (Hydra, as discussed in Section \ref{sec:res:best-method}).

Hydra expects both a robot frame and odometry as TF transforms instead of raw odometry poses. Since the real-world data was recorded using a ZED camera, it does not have a defined robot frame. This meant that during deployment, a static transform had to be published to define this frame. As well as this, a custom node, which was developed with assistance from Generative AI Claude (Anthropic) \cite{Claude2024}, justification is outlined in Section \pageref{ai-use}, had to be run to republish the odometry data as TF transforms.

\subsection{Evaluation Scripts}\label{sec:sim-eval-scripts}

For each of the metrics, the results were averaged over five runs and the Standard Deviation and Coefficient of Variation were calculated. This allows the variance of non-deterministic systems like neural networks to be measured to determine method stability.

A custom script was developed to extract ground truth information from the custom office gazebo world file. The extracted data included room centroids for room detection, object centroids and bounding boxes for object detection and intersection of union measurements, as well as object labels in order to evaluate semantic accuracy. 

Due to gazebo objects having more specific names such as ‘office\_chair\_1’, these names are normalised to the most accurate corresponding label in Hydra’s label space. For example, computer to Appliance, and wastebasket to Storage. This meant that detected and ground truth objects had the same contextual labels so the confusion matrix is accurate. Label renaming was based on substring checking for regular names, for example all labels with chair in the name was renamed to just ‘Chair’. This technique meant that the meaning of the labels was maintained.

For room detection, it is unrealistic to only accept exact centroids as true positives. Therefore, in the room detection logic, a true positive is classed as within 4m of a ground truth centroid. This threshold was chosen by taking into account the sizes of the rooms in the scene. Once a true positive is found, the ground truth room is removed from future checks. This stops multiple room nodes, like Clio’s room shattering discussed in Section \ref{sec:disc-room-seg}, from getting rewarded for over-segmentation. As well as this, the Z component of room centroid coordinates are ignored, this is because the custom office scene is a single-floor, therefore Z between centroids is negligible and can needlessly inflate the distance measurement.

The same logic from above is applied to object detection, however the threshold is lowered to within 1m of a ground truth centroid. This is because objects are on a smaller scale than rooms, therefore a smaller error is more meaningful in object terms. The same one object per ground truth is applied to objects to avoid multiple detections.

For \gls{iou} measurements, the closest ground truth object by centroid distance is found, then the \gls{iou} is calculated between the bounding boxes. This logic follows the assumption that the closest centroid is meant to be the same object. This saves time as calculating the \gls{iou} between each object and ground truth object would be computationally expensive.
